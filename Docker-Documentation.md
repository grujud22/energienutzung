## Overview

This document describes the full process of containerizing a **Spring Boot backend** with a **PostgreSQL database**, publishing the image to **GitHub Container Registry (ghcr.io)**, and automating the build process with **GitHub Actions**.

**Stack:**
- Java 21 (Eclipse Temurin JRE)
- Spring Boot
- PostgreSQL
- Docker
- GitHub Container Registry
- GitHub Actions

## The Dockerfile

The Dockerfile is stored in the `Docker/` directory and gets copied to `target/` before building.

```dockerfile
FROM eclipse-temurin:21-jre

WORKDIR /opt/application
COPY *.jar ./backend.jar
EXPOSE 8080

ENTRYPOINT ["java", "-jar", "backend.jar"]
```

**Important:** The `COPY` instruction uses `*.jar` (not `../target/*.jar`) because when building from `./target`, the `.jar` and `Dockerfile` are in the same directory. Using `../target/*.jar` causes Docker to fail since it cannot access paths outside the build context.

## Manual Docker Setup

This section describes how to run the same setup using only `docker` commands — without Docker Compose.

### Step 1 – Build the Java project

```bash
.\mvnw clean
```
Deletes the old `target/` folder and previously compiled files.

```bash
.\mvnw -DskipTests package
```
Compiles the project and packages it into a `.jar` inside `target/`. `-DskipTests` skips unit tests to speed things up.

### Step 2 – Prepare the Dockerfile

```bash
cp Docker/Dockerfile target/
```
Copies the Dockerfile into `target/` so both the `.jar` and the Dockerfile are in the same build context.

### Step 3 – Login to GitHub Container Registry

```bash
export CR_PAT=YOUR_TOKEN
echo $CR_PAT | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin
```

On Windows CMD:
```cmd
set CR_PAT=YOUR_TOKEN
echo %CR_PAT% | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin
```

### Step 4 – Build the Docker image

```bash
docker build --tag ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:latest ./target
```
Builds the image from `target/` and tags it with the full `ghcr.io` path for later pushing.

### Step 5 – Push the image to GitHub

```bash
docker push ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:latest
```
Uploads the image to GitHub Container Registry. After this it is available from any machine with access.

### Step 6 – Create the Docker network

```bash
docker network create YOUR_NETWORK_NAME
```
Creates an internal network so the backend and Postgres containers can communicate by container name instead of IP address.

### Step 7 – Start Postgres

```bash
docker run -d --name YOUR_POSTGRES_CONTAINER --network YOUR_NETWORK_NAME -e POSTGRES_USER=YOUR_GITHUB_USERNAME -e POSTGRES_PASSWORD=YOUR_DB_PASSWORD -e POSTGRES_DB=YOUR_DB_NAME -p 5432:5432 -v postgres_data:/var/lib/postgresql/data postgres:latest
```

`-v postgres_data` ensures the database persists even if the container is removed. `-d` runs it in the background.

### Step 8 – Start the Backend

```bash
docker run -d --name YOUR_BACKEND_CONTAINER --network YOUR_NETWORK_NAME -p 8080:8080 -e SPRING_DATASOURCE_URL=jdbc:postgresql://YOUR_POSTGRES_CONTAINER:5432/YOUR_DB_NAME -e SPRING_DATASOURCE_USERNAME=YOUR_GITHUB_USERNAME -e SPRING_DATASOURCE_PASSWORD=YOUR_DB_PASSWORD ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:latest
```

**Note:** The datasource URL uses `YOUR_POSTGRES_CONTAINER` (the Postgres container name) instead of `localhost`. This works because both containers are on the same `YOUR_NETWORK_NAME`.

### Verify everything is running

```bash
docker ps
```
Both `YOUR_POSTGRES_CONTAINER` and `YOUR_BACKEND_CONTAINER` should appear as `Up`. The backend is reachable at `http://localhost:8080`.

### View logs

```bash
docker logs YOUR_BACKEND_CONTAINER
docker logs YOUR_POSTGRES_CONTAINER
```

### Cleanup

```bash
docker stop YOUR_BACKEND_CONTAINER YOUR_POSTGRES_CONTAINER
docker rm YOUR_BACKEND_CONTAINER YOUR_POSTGRES_CONTAINER
docker network rm YOUR_NETWORK_NAME
```

## GitHub Container Registry

### What is ghcr.io?

GitHub Container Registry (`ghcr.io`) is GitHub's built-in Docker image registry. It allows you to store, version, and distribute Docker images linked directly to your GitHub account and repositories.

An image tag follows this pattern:
```
ghcr.io / YOUR_GITHUB_USERNAME / backend : latest
   ↑           ↑          ↑        ↑
registry   username   image name  version
```

### Creating a Personal Access Token (PAT)

1. Go to **GitHub → Settings → Developer settings**
2. Click **Personal access tokens → Tokens (classic)**
3. Click **Generate new token (classic)**
4. Give it a name (e.g. `docker-push`)
5. Check `write:packages` (this automatically includes `read:packages`)
6. Click **Generate token** and copy it immediately — it will not be shown again

### Finding your published image

Navigate to:
```
https://github.com/YOUR_GITHUB_USERNAME?tab=packages
```

Or go to your GitHub profile → **Packages** tab.

### Versioned releases

Instead of always using `:latest`, you can tag specific versions:
```bash
docker build --tag ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:1.0.0 ./target
docker push ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:1.0.0
```

## CI/CD with GitHub Actions

GitHub Actions automates the entire build and push process on every push to `main`. It runs on **GitHub's own servers** — your local Docker Desktop is not involved.

### Setting up the workflow

Create the file `.github/workflows/docker-publish.yml` in your project. If `.github/workflows/` does not exist yet, create it:

```bash
mkdir -p .github/workflows
```

Or create it directly on GitHub via **Actions → set up a workflow yourself**.

### Workflow file

```yaml
name: Build and Push Docker Image

on:
  push:
    branches:
      - main

jobs:
  build-and-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Build with Maven
        run: ./mvnw -DskipTests package

      - name: Copy Dockerfile to target
        run: cp Docker/Dockerfile target/

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: ./target
          push: true
          tags: ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:latest
```

### Required repository setting

Go to **GitHub → your repository → Settings → Actions → General → Workflow permissions** and select **Read and write permissions**, then save.

This allows GitHub Actions to use the built-in `GITHUB_TOKEN` to push to `ghcr.io` — no manual PAT needed.

### What happens on every push to main

```
Push code to main
       ↓
GitHub Actions starts on GitHub's server
       ↓
Maven builds the .jar
       ↓
Docker image is built
       ↓
Image is pushed to ghcr.io/YOUR_GITHUB_USERNAME/YOUR_IMAGE_NAME:latest
       ↓
Local Docker Desktop is never involved
```
